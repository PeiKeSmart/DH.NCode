# 元数据管理总览（Configuration）

`XCode.Configuration` 是 XCode 运行时元数据层，负责把“实体类特性 + 属性定义 + 索引定义”统一收敛成可计算的数据模型，供 CRUD、查询表达式、缓存、分片、代码生成、配置中心等模块共享。

## 核心对象关系

- `TableItem`：表级元数据（表名、连接名、字段集合、主键、索引、DataTable）
- `FieldItem` / `Field`：字段级元数据（类型、长度、可空、映射、展示信息、表达式运算）
- `ShowInOption`：字段显示位置三态解析器（List/Detail/Add/Edit/Search）
- `DbConfigProvider`：数据库配置提供者（参数表驱动、支持热更新）

## 为什么它重要

很多业务以为自己在“直接查表”，实际上是在走元数据驱动：

- `Meta.Table.FindByName(...)`
- `_.FieldName == value`（`Field` 运算符）
- `FieldExtension.Between/GroupBy/Count...`
- `Meta.ShardPolicy.Field`

元数据质量直接决定：

- SQL 生成正确性
- 动态查询与排序稳定性
- 分片策略可推导性
- 代码生成一致性

## 运行时来源

`TableItem.Create(Type)` 会读取实体类型上的特性并构建元数据：

- `[BindTable]` → 表名、连接名、DbType
- `[BindColumn]` + `[DataObjectField]` → 字段定义
- `[BindIndex]` → 索引定义
- `[Description]` / `[DisplayName]` / `[Category]` → 展示信息

并缓存到静态字典，避免重复反射。

## 元数据的“控制面”与“执行面”

- 控制面（定义）
  - 实体特性、`Model.xml` 生成代码、手工扩展属性
- 执行面（运行）
  - `TableItem/FieldItem` 参与查询表达式、缓存策略、分片推导

建议：控制面尽量通过模型文件与生成器统一，减少手工偏差。

## 与其它模块关系

- 与 `Shards`
  - `IShardPolicy.Field` 依赖 `FieldItem`
  - 查询表达式分片推导基于字段元数据
- 与 `Code`
  - 生成器读取表字段元信息决定输出特性、查询方法、缓存代码
- 与 `Cache`
  - `FieldCache` 使用字段元数据构造 group by/count
- 与 `Cube`
  - `ShowIn` 决定字段在各区域展示策略

## 推荐实践

1. 实体字段命名统一（尤其主键、时间字段、租户字段）。
2. `BindColumn` 中声明 `ItemType/ShowIn/DataScale`，让展示与运行策略可配置。
3. 大改字段前先检查 `TableItem.FindByName` 下游调用，避免运行时找不到字段。
4. 扩展字段用 `AllFields` 与 `ExtendFieldNames` 管理，不要混入数据列。
