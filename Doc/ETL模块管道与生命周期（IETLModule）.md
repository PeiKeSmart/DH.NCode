# ETL模块管道与生命周期（IETLModule）

`IETLModule` 是 `ETL` 的插件式扩展接口，用于在抽取任务的关键阶段注入业务逻辑。

它让你可以在“不改 ETL 主流程源码”的前提下，完成：
- 任务前置检查
- 批次级审计
- 错误上报与告警
- 指标补充与落盘

---

## 1. 接口方法与触发时机

| 方法 | 触发时机 | 常见用途 |
|------|----------|----------|
| `Start()` | 调度启动时 | 初始化连接、缓存、指标对象 |
| `Stop()` | 调度停止时 | 释放资源、落盘收尾 |
| `Init()` | 每轮首次初始化 | 轮次级参数准备 |
| `Processing(ctx)` | 单批开始前 | 是否放行本批（返回 `false` 可跳过） |
| `Fetched(ctx)` | 抽取完成后 | 记录抽取量、抽取耗时 |
| `Processed(ctx)` | 批次处理逻辑结束后 | 统计处理量、埋点 |
| `OnFinished(ctx)` | 批次成功结束 | 提交外部偏移、发成功事件 |
| `OnError(ctx)` | 批次/实体出错 | 分级告警、异常降噪 |

---

## 2. 与 ETL 主流程的配合点

`ETL.Process()` 中对模块的调用顺序大致是：

1. `Processing(ctx)`
2. `Fetched(ctx)`
3. `Processed(ctx)`
4. `OnFinished(ctx)`（成功路径）
5. `OnError(ctx)`（异常路径）

这意味着模块可以把“抽取阶段指标”和“处理阶段指标”分开记录。

---

## 3. 推荐实践模式

### 3.1 审计模块

在 `Fetched/OnFinished` 记录：
- 批次号
- 窗口（`Start/End/Row`）
- 处理条数与耗时

### 3.2 节流模块

在 `Processing(ctx)` 判断：
- 当前时段是否允许执行
- 下游压力是否过高

不满足条件时返回 `false`，实现软暂停。

### 3.3 告警模块

在 `OnError(ctx)` 做异常分类：
- 数据格式错误：记录并跳过
- 连接错误：触发高优先级告警

---

## 4. 并发场景注意事项

当 `MaxTask > 0` 时，多批次并发执行：

1. 模块内部共享状态需加锁或使用并发容器。
2. 不要在模块中保存“当前批次全局变量”，请使用 `ctx`。
3. 对外 IO（日志/告警）建议异步并限流。

---

## 5. 极简示例（思路）

实现一个 `AuditModule`：

- `Start`：打开审计文件
- `Fetched`：记录抓取速度
- `OnFinished`：记录处理速度与成功数
- `OnError`：记录异常摘要
- `Stop`：关闭文件句柄

这样 ETL 本体不变，审计能力即插即用。

---

## 6. 设计价值总结

`IETLModule` 相当于 ETL 的“生命周期中间件”。

- 对内：减少对 ETL 核心类的侵入式改动
- 对外：把审计、告警、节流等横切能力标准化

在长期运行的数据任务中，这种解耦通常比“在业务代码里散落日志”更稳、更可维护。
