# 分库分表临时切换（EntitySplit）

`EntitySplit` 提供了一个轻量 API：在一段代码内临时切换实体工厂的 `ConnName/TableName`，执行完成后自动还原。

```csharp
factory.Split("Db_202603", "Log_202603", () =>
{
    return Log.FindCount();
});
```

---

## 1. 实现机制

核心逻辑：

1. 进入 `Split(...)` 时保存原 `ConnName/TableName`
2. 设置为目标连接与目标表
3. 执行回调 `func()`
4. `Dispose()` 中恢复原值

通过 `IDisposable` 保证异常时也能恢复。

---

## 2. 使用场景

- 历史月表按月回溯查询
- 运维工具中手动切到某张归档表执行统计
- 临时脚本跨库数据校验

---

## 3. 线程安全风险（重点）

`EntitySplit` 改的是工厂实例上的可变属性。如果同一工厂被并发请求共享，临时切换可能互相覆盖，造成“串库/串表”。

因此它更适合：
- 单线程维护脚本
- 明确串行的后台任务
- 测试代码

不建议在高并发 Web 请求主链路直接使用。

---

## 4. 推荐替代方案

生产系统优先使用“无副作用路由”：

1. 通过 `ShardPolicy` 自动路由
2. 或显式按目标库表创建 `IEntitySession`
3. 批量操作方法传入 `session` 参数

这样不会修改共享工厂状态，线程模型更安全。

---

## 5. 安全使用建议

如果必须使用 `EntitySplit`：

- 把 `Split` 包裹在业务级互斥锁中
- 回调内只做“短小、无阻塞”操作
- 禁止 `await` 跨越 `Split` 作用域（避免恢复时机漂移）

---

## 6. 小结

`EntitySplit` 是“快而轻”的临时工具，不是高并发路由基础设施。对于线上系统，优先 `ShardPolicy + Session` 路由；`EntitySplit` 更适合工具脚本和应急诊断。
