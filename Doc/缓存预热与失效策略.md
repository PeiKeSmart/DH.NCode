# 缓存预热与失效策略

在 XCode 缓存体系中，“是否命中”决定了数据库压力与接口稳定性。本篇给出可执行的预热与失效策略，配合已有缓存专题文档使用。

## 1. 目标

- 启动后快速进入高命中状态
- 写入后保证可接受一致性
- 避免缓存抖动和惊群刷新

## 2. 启动预热建议

### 2.1 实体缓存预热

```csharp
_ = User.Meta.Cache.Entities;
_ = Role.Meta.Cache.Entities;
_ = Parameter.Meta.Cache.Entities;
```

优先预热：字典类、小表、鉴权链路必经表。

### 2.2 单对象缓存预热

```csharp
_ = User.Meta.SingleCache[1];
_ = Tenant.Meta.SingleCache[1];
```

可以只预热热点 key，而不是全量 key。

### 2.3 字段缓存预热

```csharp
_ = VisitStat.FindAllPageName();
```

适合后台筛选下拉、运营面板 TopN。

## 3. 失效策略建议

### 3.1 常规写入

默认策略即可：
- 实体缓存标记过期并异步刷新
- 单对象缓存清空/过期

### 3.2 强一致链路

关键写入后可强制清理：

```csharp
User.Meta.Cache.Clear("用户权限变更", force: true);
```

用于“改完立刻读”的权限/配置场景。

### 3.3 批量导入

批量导入时建议：
- 导入期间关闭对缓存的依赖查询
- 导入后统一清理相关缓存

## 4. 防抖与限流

- 避免每次写入都手工 `Clear(force:true)`。
- 大并发接口里不要在请求线程做复杂缓存重建。
- 可以通过任务队列延后重建，前台先使用旧值短暂过渡。

## 5. 观察指标

建议关注：

- 实体缓存命中率（Success/Total）
- 单对象缓存命中率
- 缓存刷新次数（Times）
- 数据库慢查询数量

命中率持续偏低，通常说明缓存选型或过期时间不合理。

## 6. 常见误区

- **把大表放实体缓存**：内存和刷新都容易爆。
- **全部都用单对象缓存**：key 过散会导致低命中。
- **过期时间设太短**：频繁回源导致抖动。

## 7. 推荐起步参数（经验值）

- `EntityCacheExpire`: 30~120 秒（小表）
- `SingleCacheExpire`: 30~300 秒（热点对象）
- `FieldCacheExpire`: 600 秒起步（统计类）

具体值需结合写入频率和一致性要求调优。
